# Bug 修复

## 触发条件
描述了一个 bug / 异常 / 报错 / 行为不符合预期。

## 执行流程

1. **定位问题** — 分析错误信息/现象，定位到具体代码位置
2. **说明原因** — 简要说明 root cause（1-3 句话）
3. **直接修复** — 输出修复代码，最小改动原则
4. **告知影响** — 说明改动影响范围，是否有副作用

## 不需要

- 需求拆解
- 工时预估
- Figma 解析
- 结构化 Plan
- 提效记录（除非负责人要求记录到 Bug 修复有效率表）

## 调试定位方法

1. **日志/堆栈分析** — 先看错误堆栈，定位到最近的业务代码帧
2. **状态检查** — 检查当前数据流/状态是否符合预期
3. **二分排查** — 改动范围大时通过 git bisect 或注释代码二分定位
4. **平台差异** — 仅单平台复现时，优先排查原生桥接/平台特有行为
5. **网络/数据问题** — 检查 API 返回结构是否变更，请求参数是否正确

## 原则

- **最小改动** — 只改必要的代码，不要顺手重构
- **定位 root cause** — 修上游问题，不在下游打补丁
- **说清影响** — 改动是否影响其他模块/页面
- **提供复现验证方式** — 告知如何验证修复有效

## 回归验证

修复完成后必须告知负责人：
- 复现路径（操作步骤）
- 预期修复后的行为
- 可能受影响的关联功能（需额外验证的地方）
