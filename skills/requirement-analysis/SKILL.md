# 需求拆解

## 触发条件
收到新功能 / 新页面需求（有 Figma 链接、涉及新页面/新组件/新流程）。

## 输出模板

```markdown
## 需求分析
- 功能描述：[一句话说明]
- Figma 设计稿：[Figma URL]
- 影响范围：[涉及哪些模块/文件]
- 风险点：[可能踩坑的地方]
- 依赖：[需要的 API / 第三方库 / 切图资源]

## 传统工时预估
- 预估：[X 人天]（用于 AI 提效对比基线）
- 预估明细：
  - UI 搭建：[X 天]
  - 逻辑/网络：[X 天]
  - 联调/自测：[X 天]
```

## 注意事项
- 传统工时预估必须**事前填写**，不能事后编，这是提效数据可信度的基石
- 影响范围要具体到文件/模块路径级别
- 风险点关注：复杂动画、跨模块依赖、原生插件/SDK 修改、性能敏感区域
- 如果需求不明确，先和负责人确认，不要自行假设

## 风险分类

| 风险等级 | 识别条件 | 处理方式 |
|---|---|---|
| 🔴 高 | 涉及原生插件/SDK 修改、复杂动画、性能敏感（大列表/视频/音频）| 标注在 Plan 中，预留额外联调时间 |
| 🟡 中 | 跨模块依赖、API 未就绪、设计稿不完整 | 提前和负责人确认，阻塞时切换其他任务 |
| 🟢 低 | 纯 UI 页面、已有类似参考、API 已就绪 | 正常推进 |
